# AGENTS.md


直感的に理解できてすぐ実行できるようなコメント・ソースとする


## 概要

- cron で単発実行するバッチ（時刻・起動コマンドは [docs/deploy.md](docs/deploy.md) の「前提」「実行」）
- **公開 iCal（ICS）の URL** から予定を読み、 **今日・予定がある次の日** を **1 枚の PNG** にし、**Quote/0（Dot. Image API）** で表示を更新する
- カレンダー元データは **ICS の URL**（Google に限らない）
- 送信手順・パラメータは [Control Image Content（Dot.）](https://dot.mindreset.tech/docs/service/open/image_api) を正とする（例: `POST /api/authV2/open/device/:deviceId/image`）



## 技術・構成の前提

- **OS・配置・cron・Python 導入**: [docs/deploy.md](docs/deploy.md) の「前提」と「Python 3.12 のインストール」
- **言語は Python**: **Web フレームワークは使わない**。依存は必要最小限。**責務ごとにモジュール／ファイルを分ける**
- **エントリポイント**: パッケージ `handy_calendar`（`python -m handy_calendar`）
- **実行環境**: **`.venv`** を使い実行する（版は `.python-version`）



## 設定（環境変数）

- `.env.example` をコピーして `.env` を作成する（`.env` は **Git に含めない**）
- **読み込み**: **`python-dotenv`** を利用して `.env` を読む
- **値だけ**書き換える（変数名は `.env.example` のまま）

| 変数 | 入れる値 |
|------|----------|
| `ICAL_URLS` | 公開 ICS URL。複数は **カンマ区切り** |
| `DOT_API_TOKEN` | Dot. API トークン |
| `DOT_DEVICE_ID` | Dot. デバイス ID |



## 予定の範囲（今日・次の予定日）

- **基準日時**: バッチ開始時点の **JST カレンダー日**（例は **2026 年**）


### 例

| 予定 | 開始 | 終了 |
|------|------|------|
| 終日の予定 | 5/29 終日 | 5/29 終日 |
| 前日〜今日の予定 | 5/28 23:00 | 5/29 01:00 |
| 当日の予定 | 5/29 10:00 | 5/29 11:00 |
| 今日〜明日の予定 | 5/29 23:00 | 5/30 01:00 |
| 明日の予定 | 5/30 10:00 | 5/30 11:00 |

表示例は **5/29 0:10** と **5/30 0:10**（終了済みは載せない）。


#### 基準日 5/29（0:10）

```
今日（5/29金）
　終日の予定
　前日〜今日の予定（~01:00）
　当日の予定（10:00~11:00）
　今日〜明日の予定（23:00~01:00）
明日（5/30土）
　明日の予定（10:00~11:00）
```


#### 基準日 5/30（0:10）

```
今日（5/30土）
　今日〜明日の予定（~01:00）
　明日の予定（10:00~11:00）
明日（5/31日）
```


### 実装ルール（例から読み取れない部分）

| 項目 | 内容 |
|------|------|
| 日枠 | [0:00, 翌日 0:00) JST、`DateRange.overlaps` |
| 今日枠 | 重なる timed / 終日。`reference_now` より前に終了したものは除く |
| 2枠目の日付 | 開始日（JST）> 基準日 の最小日。無ければ翌日の空枠 |
| 2枠目の予定 | 開始日がその日と一致するもののみ |
| 終了済み | バッチ開始時点で終了済みは載せない |
| 並び | URL 列挙順 → 開始時刻 → `UID` |
| RRULE | 基準日から **90 日** 展開。`RECURRENCE-ID` 例外は展開側のみ |
| 複数 URL | 全件取得・解析成功時のみ PNG / Dot へ |
| TZ | `DTSTART` / `DTEND` を JST に正規化 |



## 振る舞い（必ず守る）

- パイプラインは **iCal 取得・解析 → PNG 生成 → Dot 送信** の順。**前段が成功したあと**だけ次へ進む
- 同じ入力なら同じ PNG になるよう **出力は決定的** に寄せる
- カレンダーの「正本」を Quote/0 やこのバッチにしない（編集は各カレンダーサービス側）
- 常駐プロセスや GUI は前提にしない（**cron 等の単発実行**）

### 成功または失敗時の挙動

| 終了コード | 意味 |
|------------|------|
| `0` | 全段成功 |
| 非 `0` | いずれかの段階で失敗（cron の失敗検知用）。**Dot は更新されない**（途中失敗時は直前の表示を維持） |

- 非 `0` のとき、原因が追える事実を残す（例: `iCal 取得失敗 url=… status=…`）。**出力先（ログ／stderr 等）は未定**



## 単体テスト（最小仕様）

- テストランナーは `pytest` を使う
- 単体テストは責務単位で分ける（**iCal 取得**、**iCal 解析**、**PNG 生成**）
- `config.py` の `load_config` は必須環境変数不足で失敗し、原因が分かる日本語メッセージを返す
- テストファイルは実行ファイル単位で分ける（例: `config.py` は `tests/test_config.py`）
- テスト関数は対象関数ごとに分ける（1 テスト関数 = 1 関数の 1 観点）
- `iCal 取得` は複数 URL の全件成功時のみ成功とし、1 件でも失敗したら失敗にする
- `iCal 解析` は上記「例」と「実装ルール」を満たす
- `PNG 生成` は同一入力で同一出力になることを保証する
- **Dot 送信は単体テスト対象外** とし、必要な確認は結合テストで行う



## 表示（PNG）

- **1 枚**に **今日・次の予定日** の 2 日分を載せる（予定が無い日も **日付枠は表示** する）
- **2 枠目**も描画行に含め上から順に描く。下端はみ出して見切れてもよい
- **日付ラベル**: 1 枠目は `今日（M/D曜）`、2 枠目が翌日なら `明日（M/D曜）`、それ以外は `N日後（M/D曜）`
- **予定行**: 時刻表記は上記「例」どおり
- 解像度・余白・フォント等のピクセル仕様は **Dot / 対象デバイスの表示要件** に合わせ、実装で固定する（変更時は見た目・送信サイズへの影響を説明する）

### コメントの書き方

- **日本語**。ぱっと見で意図が分かる短さ（例・典型呼び出し・段名の役割）
- **仕様の羅列は書かない**（終了コード・環境変数などは本書 / [docs/deploy.md](docs/deploy.md)）
- 実装指示がない限り、**コードや依存を勝手に増やさない**
- **iCal 取得 → 予定抽出 → PNG 生成 → Quote/0** の段の順を崩さない
- 依存追加・描画変更・運用（cron 等）の変更は、**影響を短く説明**する
- 要件が曖昧、または意図と食い違う・違和感があるときは、**質問してから**進める
- レート制限・パラメータ名・エラーコードは **公式ドキュメントを正**とする
- タイムゾーンは **日本標準時（JST）**
